10) we are using REDIS
Redis is basically a super-fast, in-memory temporary database that you use when you need SPEED and LOW LATENCY.

Think of Redis like this:

👉 Your main database (MongoDB/PostgreSQL) = long-term storage
👉 Redis = your app’s high-speed scratchpad / cache / session store

Redis does NOT replace MongoDB.
Redis BOOSTS MongoDB.

⚡ WHY REDIS IS SO FAST

Because unlike MongoDB/Postgres, Redis stores data in RAM, not disk.

RAM = microseconds
Disk = milliseconds

That speed difference matters A LOT.

🚀 WHAT REDIS IS USED FOR (REAL-WORLD USE CASES)
1️⃣ Caching (the most common use)

Example:
Your backend hits a slow API (weather, exchange rates).
Instead of fetching every time → Redis stores the last result.
→ Your app becomes 10–50× faster.

2️⃣ Rate Limiting
Protect API from spam or too many requests.
Example logic:
Allow 100 requests per user per 1 minute
Use Redis counters to track user requests

3️⃣ Session Store (Instead of storing sessions in DB)
If you use login sessions (not JWT), Redis is perfect.

4️⃣ Storing OTPs / Tokens / Temporary Data
Redis is ideal for:
OTP logins
Password-reset token
Email verification codes
Because data expires automatically.

5️⃣ Queues / Background Jobs
Redis is used with BullMQ or Celery for:
Sending emails
Processing videos
Running heavy tasks in background

Schema design explaination
The schema design that separates user, student, and teacher entities provides several architectural benefits. Here's a detailed explanation of this design approach:

1. Core Structure:
- A base User table containing common authentication and profile fields (username, password, email, etc.)
- Separate Student and Teacher tables with role-specific attributes, each linked to the User table via foreign key relationships

2. Implementation Details:
- The User table serves as the central identity provider
- Student table contains academic-specific fields (grade level, enrollment date, GPA)
- Teacher table includes professional attributes (department, hire date, qualifications)
- Relationships are established through one-to-one mappings between User and each role table

3. Key Benefits:
a) Data Integrity:
- Prevents null fields by separating role-specific attributes
- Maintains referential integrity through proper foreign key constraints

b) Security:
- Enables role-based access control at the database level
- Allows different permission schemes for student vs teacher data

c) Performance:
- Optimizes queries by avoiding sparse tables with many null values
- Reduces row size for common user lookups

d) Extensibility:
- Simplifies adding new role types in the future
- Supports polymorphic relationships where needed

4. Schema Validation Requirements:
- Every Student/Teacher must reference exactly one User record
- User records should validate required fields before role assignment
- Appropriate indexes should be created on all foreign keys

5. Expected Outcomes:
- Clean separation of concerns between authentication and role data
- Efficient querying for both user management and role-specific operations
- Scalable foundation for adding future role types or attributes
